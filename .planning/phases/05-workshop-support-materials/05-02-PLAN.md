Create CONTEXT-ENGINEERING.md with the following sections:

**1. Introduction**
- What is context engineering for AI agents?
- Why the choice between tools/RAG/sessions matters
- How this document helps you make architectural decisions

**2. The Core Question**
Center the document around: "What kind of data is this, and when does it change?"

**3. Quick Decision Table**
```markdown
| Question | Tool (Function Calling) | RAG (Knowledge Retrieval) | Session State |
|----------|------------------------|---------------------------|---------------|
| Does data change in real-time? | YES | NO | NO |
| Is it external API data? | YES | NO | NO |
| Is it pre-indexed documents? | NO | YES | NO |
| Is it user-specific preference? | NO | NO | YES |
| Example: Flight prices | Tool | - | - |
| Example: Visa requirements | - | RAG | - |
| Example: User budget | - | - | Session |
```

**4. Decision Flowchart**
ASCII flowchart (from research):
```
User Query
    |
    v
Does this need REAL-TIME data?
(prices, availability, current status)
    |
   YES --> Use FUNCTION CALLING TOOLS
    |       (search_flights, search_hotels)
    |
   NO
    |
    v
Is this STATIC KNOWLEDGE?
(guides, policies, cultural info)
    |
   YES --> Use RAG RETRIEVAL
    |       (destination_knowledge)
    |
   NO
    |
    v
Is this USER-SPECIFIC preference?
(budget, style, dietary)
    |
   YES --> Use SESSION STATE
    |       (user: prefix)
    |
   NO --> LLM general knowledge
```

**5. Detailed Comparison**
| Aspect | Function Calling | RAG | Session State |
|--------|-----------------|-----|---------------|
| Data freshness | Real-time | At indexing time | User sets |
| Latency | API call time | Vector search time | Instant |
| Cost | Per API call | Per query | Free |
| Setup complexity | Define tools | Index corpus | State prefixes |
| When to update | Never (live) | Re-index | User action |

**6. Workshop Examples**
Map each workshop exercise to the decision:
- Exercise 2 (search_flights): Real-time flight data -> Tool
- Exercise 3 (destination guides): Static knowledge -> RAG
- Exercise 4 (user budget): User preference -> Session

**7. Common Mistakes**
❌ Using RAG for real-time data
❌ Using tools for static knowledge
❌ Using session state for data that changes

**8. Post-Workshop: Applying to Your Projects**
Questions to ask when designing your agent:
1. List all data sources your agent needs
2. For each: does it change while user talks?
3. For each: is it per-user or shared?
4. Apply the decision framework

**9. Hybrid Patterns**
When you need multiple approaches (like our travel agent):
- ADK constraint: RAG cannot mix with function tools in same agent
- Solution: Hybrid coordinator pattern (reference hybrid_agent.py)
- When to consider: comprehensive assistants needing tools + knowledge
